# PROJECT CONSTITUTION

本文件是 `xtgzpt` 协同工作平台的项目宪法。后续开发、审查、测试、验收和 Loop 执行必须先遵守本文件。

## 1. 文档地位

本文件用于把已冻结规则、已落地技术栈、当前实现状态和后续生产标准收口到一个入口。

冲突处理顺序：

1. 本文件的边界和禁止项优先。
2. `docs/01_GOVERNANCE_FREEZE_G000_G019.md` 的冻结规则优先于普通 backlog。
3. `docs/dev-readiness/*` 是实施细化文件，不得推翻本文件。
4. `docs/dev-log/*` 记录历史事实，不能用历史阶段口径推翻当前宪法。

## 2. 项目定位

`xtgzpt` 是企业内部协同工作平台。

核心定位：

- 项目协同
- 任务协同
- 聊天协同
- 知识沉淀和问答
- 合同风险审查
- 人工审批
- 权限、审计和证据链严格的内部系统
- AI 辅助分析、整理、提醒和草稿生成

非定位：

- ERP
- 财务系统
- 采购系统
- 库存系统
- 销售系统
- 资产系统
- 报表经营大屏
- 外部客户门户
- AI 自动办公机器人

## 5. 第一版角色标准

第一版必须支持 11 类角色。当前代码只落地了 4 类基础角色，这是阶段性差距，不能标记为角色体系完成。

必须支持：

| 角色 | 用途 |
| --- | --- |
| 普通员工 | 发起、参与、处理自己相关工作 |
| 项目负责人 | 管理授权项目、任务、成员和风险 |
| 部门负责人 | 查看和处理本部门相关工作 |
| 管理层 | 查看授权范围内整体工作进度和关键风险 |
| 合同发起人 | 发起合同上传、审查和审批 |
| 法务审批人 | 审查合同风险和合同审批节点 |
| 财务审批人 | 参与合同或费用相关审批，不等于财务系统 |
| 审批人 | 根据流程节点进行人工审批 |
| 知识管理员 | 审核和发布知识内容 |
| 系统管理员 | 配置组织、角色、权限、流程和 AI 框架 |
| 超级管理员 | 管理系统级配置和管理员权限 |

原则：

- AI 不是角色。
- 系统管理员不默认拥有全部业务数据。
- 角色必须能通过后台配置扩展，不应写死到只能支持 4 类。

## 6. 权限维度标准

第一版权限必须覆盖 6 个独立维度：

1. 菜单权限
2. 数据权限
3. 操作权限
4. 审批权限
5. 文件权限
6. AI 权限

当前代码已覆盖菜单、数据、操作、文件、AI；审批权限仍混在操作权限中，这是阶段性差距。

后续必须补：

- 独立 `approval` 权限维度。
- 流程模板权限。
- 节点审批人权限。
- 转交、加签、退回权限。
- 审批代理和授权范围。
- 审批结果写回来源对象的权限和审计。

## 7. AI 边界

AI 只能做：

- 分析
- 总结
- 提醒
- 建议
- 风险提示
- 草稿生成
- 权限过滤后的知识检索

AI 严禁：

- 自动审批
- 自动驳回
- 自动发布知识
- 自动创建正式任务
- 自动签署合同
- 自动付款
- 自动确认收货
- 自动确认执行完成
- 自动改变正式状态
- 绕过人工确认
- 读取用户无权访问的来源

当前实现必须保持：

- AI 输出默认是草稿或建议。
- 人工确认后才允许生成正式任务、知识或项目记忆。
- 每次 AI 输出必须记录 `frameworkVersion`。
- 后续生产实现必须补全 `frameworkId`、输入来源、证据引用、采纳/驳回/修改记录和 AI Run 审计。

## 8. 数据与生产持久化标准

当前阶段大量数据仍是内存存储，这是开发阶段事实，不是生产标准。

生产标准必须满足：

- 项目、任务、聊天、消息、AI 草稿、知识、项目记忆、合同、审批、文件、审计全部进入数据库。
- 数据库迁移必须可重复执行、可回滚或有补偿迁移。
- 关键对象必须有 `id`、组织归属、状态、创建人、创建时间、更新时间。
- 正式状态变化必须有状态历史。
- 审计日志不可随意删除。
- 文件只能归档或作废，不允许正式流程文件物理删除。
- API 不得直接操作内存数组作为生产数据源。

生产数据库目标：

- 使用关系型数据库作为主数据源。
- 当前技术标准采用 PostgreSQL 兼容方案。
- 后续如更换数据库，必须先更新技术标准和迁移策略。

## 9. 审计标准

所有关键动作必须审计，包括：

- 登录成功和失败
- 权限拒绝
- 项目创建、成员变更、状态变化
- 任务创建、分配、提交、确认、取消、归档
- 聊天创建、消息发送、AI 整理
- AI 草稿生成、人工确认、入库
- 知识检索、知识发布、知识归档
- 合同上传、粘贴、AI 审查、风险确认、二次审查、审批、执行跟踪
- 审批发起、通过、驳回、退回、转交、加签
- 文件上传、预览、下载、归档、AI 引用
- 权限、流程、角色、组织、AI 框架配置变更

任一审计记录必须能回答：

- 谁
- 在什么时候
- 对哪个对象
- 做了什么
- 结果是什么
- 原因是什么
- request id 是什么
- 是否涉及 AI
- AI 框架版本是什么
